## SNP quality

Each SNP has several statistics which measure aspects of the likelihood that it was called correctly. The two things I focus on are mapping quality and read depth. 

The likelihood that the variant calling algorithm puts that the called SNP is correct is measured by a mapping quality score. I want to filter out all variants with a low mapping quality. 

Read depth measures the number of reads which aligned to a site. It should be fairly even across the genome. Regions where the read depth differs from the baseline value are more likely to have erroneous SNPs - due to misassembly of the reference genome or too few reads to correctly infer heterozygosity when it is present, or excess reads inaccurately mapped to the site and introducing spurious "variation". 

To do effective filtering, I need to figure out what read depth and mapping quality I should generally expect from this dataset. Most SNPs should be fairly reliable. SNPs with outlier values for these two statistics are the sus ones which I'll want to filter out.

## Contig quality

The genome is comprised of 241 supercontigs. The largest 28 of these contain half of the bases in the genome. The other 213 are, logically, much smaller. Some of them are quite small indeed. The small supercontigs are small because they just haven't been matched up with thier neighbour sequences. However, for some of them the reason why they haven't been able to assemble into a larger super-contig might be that the small supercontig is missassembled and doesn't represent anything in the genome, or because it is a duplicate of a sequence elsewhere in the genome. Thus, I hypothesise that the smaller supercontigs are more likely to have dodgy assemblies. This might affect SNP quality. I don't know. # PLINK pgen and bed files

Several downstream analyses cannot handle VCF files and instead expect .bed or the newer .pgen file types, which are produced by PLINK. The latest version, PLINK2, uses .pgen files which contain more information, but many applications need the older .bed format. I create both. 

PLINK has an automatic assumption that variants exist in fully assembled chromosomes numbered 1,2,... Instead, I have supercontigs named Scaffold1_Amanita_muscaria, etc. The tag --allow-extra-chr causes PLINK to accept which aren't numbered.  

```
plink2 --vcf ../2.filtration_levels/bcffilter1.vcf.bgz --allow-extra-chr --make-pgen --out filter1
plink2 --pfile filter1 --allow-extra-chr --make-bed --out filter1
```
